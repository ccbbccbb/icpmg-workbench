# AGENTS.md

## general
- do not edit this file unless explicitly asked to by the user

## code hygiene
- always use existing components + their styling. if a new component needs to be created, consider creating a variant (cva+cn) or adding props on an existing component.
- use React Query for client server-state reads; keep UI, animation, wallet prompt, and transaction phase state local
- remove dead code once it is clearly obsolete
- if removal is not yet certain, comment it out and mark it with `TODO(legacy:context)` with a short note

## planning & checklists
- multi-step plans and optimization/refactor docs must use checklist-driven execution with `[ ]` for todo and `[X]` for completed & verified work
- do not mark a checklist item `[X]` until the corresponding code exists
- do not make edits to SKILL.md files and surrounding files unless explicitly asked to

## testing philosophy
- tests lock important behavior; they are not required for every change
- for client code, add focused unit tests when they cheaply protect serious game-breaking invariants that would otherwise need browser/e2e coverage, such as token normalization, route/storage handoffs, tx/event parsing, or state-machine gates
- skip tests for low-risk UI/display changes when existing validations are enough; briefly note why no test was added for regression fixes
- be conservative with Cairo/sozo tests until the team settles a stronger policy. add or update them only for critical game rules, auth, economics, randomness, storage/schema invariants, battle outcomes, or serious regressions
- prefer tightening an existing focused test over adding large new fixtures; ask before expanding the contract suite when value is unclear

## conventional commits
- all commit messages must follow conventional commits formatting: `<type>: <description>`
- required: type and description
- optional: body/footer after a blank line
- `feat`: features
- `fix`: bug fixes
- `chore`: maintenance
- `patch`: small non-feature code updates
- `docs`: documentation changes
- `refactor`: code restructuring without behavior changes
- `ci`: continuous integration workflow changes
- `style`: formatting-only changes
- `test`: adding or updating tests

## git
- do not stage, unstage, or make commits unless explicitly asked to
- do not lecture the user on the repositories uncommited/unstaged/staged files - assume they are monitoring the situation closely & if they require assistance with git, they will ask 
- never force push
